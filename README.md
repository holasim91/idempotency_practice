# 멱등성(Idempotency)

## 배경

전에 클라이언트에서 동시성을 LWW로 처리하는 예제를 만들었다.
예제를 마치면서 비동기 요청을 처리할 수 있는 다른 방법이 뭐가 있을까 생각 하다가 모 홈페이지에서 파일을 첨부할때 파일이 업로드 되는 동안 여러번 누르면 같은 파일이 중복으로 올라가는 현상이 생각났다.
혼자 생각도 해보고 AI에게도 의견을 물어봤는데 꽤나 많은 방법들이 있었지만 가장 눈에 띄었던건 **Idempotency key** 방식이었다.
Idempotency? 처음 보는 단어라 검색을 해봤더니 **멱등성** 이라는 뜻의 단어였다.  

멱등성!!  

멱등성이라는 단어를 여러번 들어봤지만 정작 그게 무엇인지 찾아본적이 없었다.
**멱등**하다는 것이란 **여러번 무언가 수행을 해도 그것을 한 번 수행한 것과 결과가 같다** 라는 뜻.
그래서 이번 기회에 반복되거나 동시에 들어오는 동일한 요청을 멱등하게 처리하는 방법을 학습해보려고 한다.
이번에는 서버까지 직접 구현해볼 생각이다.

## 문제 정의

사용자가 파일 하나를 선택한 뒤 업로드 버튼을 빠르게 세 번 누른다.

의도:

- 하나의 파일을 한 번 업로드

실제 결과:

- 같은 파일이 서로 다른 ID를 가진 파일 레코드 세 개로 저장

원인:

- 요청 중 버튼이 활성화되어 있음
- 서버가 각 요청을 독립적인 새 업로드로 처리
- 동일한 작업인지 식별하는 기준이 없음

## 멱등성이란

여러번 요청을 해도 한 번 요청한 것과 결과가 같다는 것.

## 목표

동일한 파일 업로드 작업이 반복 요청됐을 때 중복 파일이 생성되는 문제를 재현하고, 클라이언트 방어와 서버 측 멱등성 보장의 차이를 확인한다.

## 실습 환경

- FE: Vite+React  
- BE: Express

## 1. 중복 업로드 문제 재현

구현: 가짜 업로드(1.5초 Promise)로 서버 왕복을 흉내, 레코드 배열 = 가짜 DB  
재현 결과: 파일 1개 선택 → 3연타 → 임의의 파일 레코드 3개, 각기 다른 UUID (문제 정의 그대로 재현됨)

## 2. isUploading을 이용한 클라이언트 방어

구현: isUploading state + disabled + 버튼 텍스트 변경  
결과: 업로드 연타 → 레코드 1개, 방어 성공

## 3. 클라이언트 방어의 한계

클라이언트 만으로는 유저의 중복요청을 완벽히 막는 것을 보장 못함.
API로 직접 요청을 한다던가, 서버의 응답이 오기전에 새로고침을 한다던가, 서버에서의 응답이 아예 유실된다던가.
따라서 요청이 모이는 서버에서도 중복요청을 방어하는 처리가 필요하다는 결론이 나옴.

### 생각해 볼 것

- 클라이언트의 중복요청을 막기 위한 플래그로 지정된 state는 브라우저의 탭 한 곳에서만 살아있음.
- 그러므로 다른 탭, 다른 기기 등에서 동일한 요청을 할 수 있음.
- 그런데 이 행위를 중복요청으로 봐야할지는 **유저의 의도** 수준에서 생각해 볼 문제.(막을 대상인가?에 대한 고민)
- 따라서 동일한 파일을 다시 올릴 때 '이미 업로드된 파일입니다'라고 확인받는 UX로 사용자의 의도를 되묻는 방식으로 처리해야하지 않을까.

## 4. Idempotency Key를 이용한 서버 방어

### 구현 흐름

클라이언트: 파일 선택시 멱등키 생성 -> state로 보관 -> 서버 요청시 헤더에 담아 전송
서버: 헤더의 멱등키 확인 -> 헤더 키 부재 ->400에러(키 없음)
                   -> 헤더 키 존재 -> Map으로 만든 임시 DB에서 키 등록 유무 조회 -> 키 존재 -> 등록된 데이터 반환
                                                                     -> 키 부재 -> 새로 등록(응답시간 1.5초) -> 등록중 다시 요청(status:pending) -> 409 Confilict

### 결과

- 같은 키 동시 3번 → 레코드 1개 (1× 200, 2× 409)
- 완료된 키(completed) 재시도 → 새 저장 없이 첫 결과와 동일한 id 반환

#### NG - 동시요청

- 문제: 순차적으로 오는 중복요청은 가능했으나 동시에 시도하는 요청은 방어하지 못함.
- 원인: 조회와 기록 사이에 1.5초짜리 비동기 처리가 통째로 끼어 있어서, 그동안 도착한 요청은 모두 '처음 보는 키'로 판정됨
- 1차 해결: isPending이라는 전역 플래그를 만들고 set하기 전에 pending중이라는 상태로 바꿔서 방어 시도.
- 요청들이 키가 없는 상태로 로직에 들어왔기 때문에 해당 요청이 등록중인지 판별하는 로직 자체에 접근을 하지 못함.
- 게다가 전역으로 선언을 했기 때문에 다른 요청의 상태에 오염 유발 가능성이 있음. 따라서 실패
- 2차 해결: Map에 값을 등록을 할 때 상태도 같이 추가. 그리고 비동기요청이 일어나기 전으로 이동.
- 이러면 다음 요청은 DB를 조회하는 로직에 걸리고, 해결.
- 이유: 내 생각은 조회 -> 1.5초 -> 생성 or not이라고 생각을 했는데, 그 생각이 잘못된 것. 그 사이에 도착한 동시 요청들은 모두 키가 없는 상태로 판정이 되어서 통과가 되어서 요청한 만큼 레코드가 생기게된다. 그리고 비동기 처리 전으로 데이터를 등록하게 되면 조회와 생성 사이에 틈이 없어지기 때문에 첫 요청 다음의 요청은 자연스럽게 테이블 조회 분기를 타게 된다.(Node는 싱글 스레드이고 제어권은 await에서만 넘어간다. get과 set 사이에 await이 없으면 그 구간은 끊기지 않으므로 다른 요청이 끼어들 수 없다)

### 인메모리에서 DB로 확장

#### 구현 흐름 - DB추가

서버: 헤더의 멱등키 확인 - 헤더 키 부재 -> 400에러(키 없음)
          |
      헤더 키 존재 -> DB에 등록 시도(상태와 멱등키만 insert)(응답시간 1.5초) - 등록 됨 -> status, record 업데이트
                            |
                           안됨 -> 조회 -> status === 'pending' -> 409 Conflict
                                      -> status === 'completed' -> 등록된 데이터 반환

- 직전에 했던 실습은 다시 실행을 했을때 모든 메모리가 날아가기도 하고, 만약 인스턴스가 여러개인 상황에서는 무용지물.
- 멱등성을 유지해야한다는 본질은 같지만 데이터의 멱등성을 판단하는 위치를 인메모리(싱글스레드의 원자성)에서 DB(PK의 원자성)로 바꾸었다.
- 그 덕에 인스턴스가 여러개 있는 환경에서도 멱등성을 유지할 수 있는 환경을 만들 수 있게됐다.

#### NG

- 처음엔 Map에서 구현한 흐름을 그대로 바꾸려고해서 SELECT 후 INSERT로 바꿨는데, 생각해보니 멱등키를 PK로 설정해놨으니 INSERT 할 때 자동으로 조회까지 하게 된다는 것(검사 + 생성의 원자성)을 깨달았다. 그래서 바로 INSERT문으로 조회와 생성까지 한 번에 처리를 하게 했다.(INSERT First)
